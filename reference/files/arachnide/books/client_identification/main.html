The Client Identifaction Mechanism



  • First to dash forward: this here is just a recommandation, not a mandotory or anything defined by an 'offical' institute.
    It's a system introduced by Arachnide and I hope current clients will turn over to join it, and of course that all future one's will respect it.

  • So what it's all about?
    Well the idea is to give the server a chance to know which clients are connecting to it. This can have several benifits:
    • Statistik purposes.
      A server admin can gain statistics about the clients which his users use.

    • Protocol enhancement.
      A server could possible be able to talk an enhanced protocoll with 'friend' clients, which are capable to do so also. So if a 3rd party clients connects to a server, it can check if it's a friend client and if there is a matching enhancement protocol. The client will realize the protocol chosen from the server in the first answer. (details have to be defined, when necessary)

    • Client denial.
      Well it doesn't sound too attractive for client developers in first place, but it can be quite an enhancement for server administration. Unsupported clients with known bugs which are a thread to server stability can so simply be rejected. This can also be used to reject old client versions, if there is a newer ver. aviable, where some issues are fixed.


  • Technical
    Currently on every connection the first 4 bytes send by the client are the encryption seed.
    A server emulator-3rd party client has never an intention to encrypt data, as this is only a waste on processor performance. As 3rd party talk unencrypted this seed can be used for the client identification mechanism.

    The OSI client sends always the client IP as encryption seed.
    In example: 47.11.8.15
    As the numbers 0 and 255 aren't legal IP adresses they can be used to seperate the OSI client from 3rd party clients.

    Third party clients send 0 as very first byte.
    • The numbers 1 .. 254 are used by the OSI client.
    • 255 can be used for future issues.
    • I've heared that 15 has a special meaning, don't know exactly. (15.*.*.* is the HP net, so anyone working there can't connect :o)
    • Some servers (sphere) also take away 32 for it's console, but thats not matter of this recommandation.

    Byte 2 identifies the client See the table below which numbers are associated to which clients.
    Byte 3 and 4 allow to identify the client version.
    The most self-evident and recommended is MAJOR.MINOR
    So to summarize it up, the first four bytes are:

      0.CLIENT.MAJOR.MINOR
